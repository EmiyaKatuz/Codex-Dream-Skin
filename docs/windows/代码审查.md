# Windows 代码审查记录

## 2026-07-23 01:06

### 审查范围

- `windows/assets/renderer-inject.js`：DOM 分类、主题重注入、观察器、定时器、布局测量和异常边界。
- `windows/assets/dream-skin.css`：动画、滤镜、`backdrop-filter`、`:has()` 和 reduced-motion 降级。
- `windows/scripts/injector.mjs`：CDP WebSocket、命令超时、watch 模式、早期注入、重载回退和进程信号。
- `windows/scripts/tray-dream-skin.ps1`：托盘菜单重建、事件异常和 WinForms 资源释放。
- Windows 测试入口及相关 Node 回归测试。

### 结论

发现并修正 6 组可复现或具备明确触发链的缺陷。其中 2 组会显著放大流式输出期间的 CPU/布局开销，2 组可能造成注入器退出或长期连接泄漏，2 组会造成超时失效、重复监听或 WinForms 资源滞留。本次修复发布版本更新为 `1.3.5`。

### 已修正问题

#### [P1] 流式 DOM 变更可触发高频全页面布局扫描

- 触发链：`MutationObserver` 在一次动画帧内合并变更，但连续流式输出可以让 `ensure()` 接近逐帧执行；原实现每次都遍历 `body *`，并对所有元素调用 `getBoundingClientRect()` 与 `getComputedStyle()`。
- 影响：强制样式与布局计算随页面节点数增长，容易造成输入卡顿、滚动掉帧、CPU 占用升高；长会话和高频输出下风险更明显。
- 修正：为 mutation 刷新增加 120ms 最小间隔；resize/motion 仍可请求即时刷新；全文本候选先进行短文本匹配，仅对少量命中节点做可见性测量；所有触发源共用一个可取消调度器。
- 回归：加入突发 mutation 只排队一次刷新，以及 1000 个无关流式文本节点产生 0 次候选样式读取的测试。

#### [P1] CDP 会话初始化失败和监听器异常可能终止注入器

- 触发链：WebSocket 已连接后，`Runtime.enable` 或 `Page.enable` 失败会让 `open()` 拒绝；调用方尚未取得 session 引用，旧实现无法关闭半连接。CDP 事件监听器的同步异常也会越过消息回调边界。
- 影响：watch 重试时遗留 WebSocket；监听器异常可成为未捕获异常并结束注入器进程，主题随后停止维护。
- 修正：`open()` 对握手与 domain enable 使用统一异常清理；启动监听器在 settle 后解除；同步及异步监听器错误分别隔离并记录；`close()` 幂等释放 pending command 与 listener closure。
- 回归：新增 `injector-session.test.mjs`，覆盖 domain enable 失败关闭、监听器隔离、取消订阅、幂等清理。

#### [P2] 暂停/恢复后 load fallback 监听可能重复累积

- 触发链：暂停时旧代码删除 `fallbackListeners` 的登记，但实际 `Page.loadEventFired` 监听仍挂在 session 上；恢复后某次 early-script 注册失败会再次添加监听。
- 影响：多次暂停/恢复后，一次页面加载可能并发执行多次完整 payload，增加 CPU、内存和界面闪烁风险。
- 修正：监听登记改为可取消的 Map；暂停期间保留当前 session 的唯一监听；同一 target 的连续 load 事件合并为一个 250ms 回退任务；target 关闭、失败和 watcher 退出时同步取消 timer 与 listener。
- 回归：bootstrap 测试验证暂停分支不会遗忘仍挂载的 listener，并验证回退 timer 合并与 signal listener 清理。

#### [P2] 调用方指定的 CDP 超时没有传递到命令层

- 触发链：`updateOperationUi()` 向 `session.evaluate()` 传入 1500/3000ms 等超时，Windows `evaluate()` 原签名忽略第二个参数，底层始终等待 10000ms。
- 影响：托盘应用、暂停或结果提示在连接异常时可能额外冻结数秒，快速失败语义失效。
- 修正：`send()` 与 `evaluate()` 接受并传递 `timeoutMs`。
- 回归：session 生命周期测试断言 275ms 自定义值到达底层 CDP 命令。

#### [P2] 重注入会连续写入两次完整 CSS

- 触发链：复用旧 style 时先写入 CSS 并标记 revision `3`；随后 `ensure()` 期望 revision `8`，立即再次写入同一份约数百 KB 的样式表。
- 影响：每次重应用或切换主题会重复解析样式，并可能放大首帧抖动。
- 修正：使用单一 `STYLE_REVISION` 常量，复用与校验路径保持同一 revision。
- 回归：renderer 测试锁定统一 revision，并禁止旧 revision 写入路径回归。

#### [P2] ResizeObserver 与托盘菜单资源释放不完整

- 触发链：主壳移除时，旧 resize target 仍留在 Set 与 observer；托盘每次打开菜单时只调用 `Items.Clear()`，旧 ToolStripItem 未主动释放，Opening 事件异常也缺少边界。
- 影响：页面重建可能短期保留旧 DOM；频繁打开托盘菜单会延迟释放菜单项及其闭包；菜单数据异常可能结束托盘消息循环。
- 修正：清理皮肤时逐个 unobserve 并清空 target；重建菜单前 Dispose 旧项；Opening 事件捕获错误并取消打开；退出时解除并 Dispose ContextMenuStrip、NotifyIcon 和 Icon。
- 回归：PowerShell 总测试验证托盘资源清理与错误边界所需语句，并继续执行真实 WinForms helper 检查。

### 保留风险与后续建议

- 大型 CSS 仍包含较多 `:has()`、`backdrop-filter` 和滤镜规则，用于保证 Codex 动态面板首帧覆盖。当前动画已提供 reduced-motion 降级；建议发布后使用实际长会话继续采集 Renderer CPU、Layout 和 GPU memory 数据，再依据热点缩小结构回退选择器。
- 本次修改集中在 Windows 主题链路；macOS 文件没有同步改动。
- 仓库 payload 尚未覆盖本机已安装的较旧运行时。本次结论来自仓库测试与静态/行为回归；发布前可在手动更新本机运行时后补一次真实 Codex 长会话压力验证。
